### 2. Pipeline Bugs Found & Fixed

#### Bug 1: Wrong webhook path
- **Symptom**: Phase 3 showed `plan_type='standard'` instead of `premium`
- **Root cause**: Test was hitting `/webhook/jotform-hvac-onboarding` (Standard)
- **Fix**: Premium uses `/webhook/jotform-hvac-premium-onboarding`

#### Bug 2: Premium call processor never worked (pre-existing bug)
- **Symptom**: `STQ4Gt3rH8ptlvMi` always returned exec=error with empty runData
- **Root cause**: Used `n8n-nodes-base.filter` node type — incompatible with this n8n version (causes silent crash before any node runs). Had NEVER worked in production.
- **Fix**: Rebuilt call processor from Standard base using `n8n-nodes-base.if` for filter

#### Bug 3: GPT returns nested JSON (Premium-specific)
- **Symptom**: caller_name, score, service all empty despite workflow succeeding
- **Root cause**: Premium GPT prompt uses section headers (`CALLER INFORMATION:`, `LEAD QUALIFICATION:`, etc.) → GPT mirrors these as nested JSON keys → `...leadData` spread doesn't flatten
- **Fix**: Added section-header flattener to Parse Lead Data using `ci`, `cc`, `lq`, `ad` shorthand

#### Bug 4: Code node char limit (~1100 chars)
- **Symptom**: Parse Lead Data updates >1100 chars caused exec=error
- **Root cause**: n8n Railway instance has JS compilation limit on code nodes
- **Fix**: Kept Parse Lead Data terse (<1100 chars) using compact variable names
